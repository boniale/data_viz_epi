1. Session goals

By the end of this session, participants will be able to:

  1. Choose a chart type that fits the analytical question.
  2. Improve a basic plot using order, labels, colour, and annotation.
  3. Apply core design principles to reduce visual noise.
  4. Turn a graph into a clear data story.
  5. Render slides with executable R code in Quarto.

4. Why humans understand visuals quickly

The brain detects:

  • Position.
  • Colour differences.
  • Movement.
  • Shape.
  • Contrast.

much faster than reading tables.

5. Data visualization is communication

3. Data visualization is communication

A good visualization answers three questions:

  1. What is the question?
  2. What should the audience notice?
  3. What action or interpretation should follow?

Note

Start with the message, not the chart.

4. The audience-message-evidence triangle

A strong data story connects:

  • Audience: Who needs this information?
  • Message: What is the main point?
  • Evidence: Which data supports the point?

Weak message:

Here is a plot of cases by week.

Stronger message:

Cases increased rapidly in the North district, peaking in week 7.

6. Bad graphs bingo

Which problems do you notice?

  • Too many colours?
  • 3D effects?
  • Tiny labels?
  • No message?
  • Bad axis scaling?
  • Too much information?

Warning

A graph can be technically correct but still communicate poorly.

10. First look: ask before plotting

Before making a chart, ask:

  1. Are we comparing groups?
  2. Are we showing change over time?
  3. Are we showing a distribution?
  4. Are we showing a relationship?
  5. Are we trying to persuade, explain, or explore?

11. Exploratory vs explanatory plots

Exploratory Explanatory
Used during analysis Used for communication
Many variables Focused message
Dense information Simplified design
Analyst-focused Audience-focused

14. Visual encoding

Charts work because visual properties encode data.

Data feature Common encoding
Quantity Position, length, area
Category Colour, shape, position
Time Position on x-axis
Uncertainty Interval, band, transparency
Emphasis Colour, annotation, size

Use the most accurate encoding for the task: position and length are usually easier to compare than area or colour intensity.

15. The hierarchy of visual perception

Humans compare these most accurately:

  1. Position.
  2. Length.
  3. Direction.
  4. Angle.
  5. Area.
  6. Volume.
  7. Colour intensity.

This is why pie charts are often difficult to compare.

16. Principle 1: Choose the right chart

Task Useful chart
Compare categories Bar chart
Show change over time Line chart
Show distribution Histogram or boxplot
Show relationship Scatter plot
Show composition Stacked bar chart, with caution

17. Common visualization mistakes

Avoid:

  • Rainbow colour palettes.
  • Pie charts with many slices.
  • Overloaded dashboards.
  • Tiny fonts.
  • Truncated axes without explanation.
  • Decorative backgrounds.

18. Practical 1: compare totals

Create a bar chart of total cases by district.

cases |>
  summarise(total_cases = sum(cases), .by = district) |>
  ggplot(aes(x = district, y = total_cases)) +
  geom_col() +
  labs(
    title = "Total cases by district",
    x = "District",
    y = "Total cases"
  ) +
  theme_minimal(base_size = 14)

19. Improve comparison with ordering

cases |>
  summarise(total_cases = sum(cases), .by = district) |>
  mutate(district = fct_reorder(district, total_cases)) |>
  ggplot(aes(x = district, y = total_cases)) +
  geom_col(fill = "#00507F") +
  coord_flip() +
  labs(
    title = "North district reported the highest total case burden",
    x = NULL,
    y = "Total cases"
  ) +
  theme_minimal(base_size = 14)

21. Principle 2: Reduce visual noise

Remove anything that does not help the audience understand the message.

Common improvements:

  • Use clear labels.
  • Avoid unnecessary gridlines.
  • Use colour with purpose.
  • Put the main message in the title.
  • Avoid 3D effects and decorative elements.

22. Cleaner time-series design

cases |>
  ggplot(aes(x = week, y = cases, color = district)) +
  geom_line(linewidth = 1.2) +
  geom_point(size = 2) +
  scale_x_continuous(breaks = 1:12) +
  labs(
    title = "North district had the sharpest rise in weekly cases",
    subtitle = "Weekly case counts across three districts",
    x = "Epidemiological week",
    y = "Number of cases",
    color = "District"
  ) +
  theme_minimal(base_size = 14) +
  theme(
    legend.position = "bottom",
    panel.grid.minor = element_blank()
  )

23. Small design changes matter

Small improvements can dramatically improve readability:

  • Better spacing.
  • Fewer colours.
  • Better contrast.
  • Better labels.
  • Simpler legends.

Good visualization is usually about subtraction, not addition.

24. Principle 3: Use colour intentionally

Colour should help the message, not decorate the plot.

Use colour to:

  • Highlight the key group.
  • Separate meaningful categories.
  • Show intensity or magnitude.

Avoid using too many colours when one highlight colour is enough.

25. Accessibility and colour blindness

Good visuals should remain readable for everyone.

Recommendations:

  • Avoid red-green combinations.
  • Use contrast.
  • Use labels, not colour alone.
  • Test plots in grayscale.

Useful package:

colorBlindness::displayAllColors()

26. Highlight the story

cases |>
  mutate(is_focus = district == "North") |>
  ggplot(aes(x = week, y = cases, group = district)) +
  geom_line(aes(color = is_focus), linewidth = 1.3) +
  geom_point(aes(color = is_focus), size = 2) +
  scale_color_manual(
    values = c("TRUE" = "#3F9C35", "FALSE" = "grey75"),
    guide = "none"
  ) +
  labs(
    title = "Cases peaked earlier in the North district",
    subtitle = "Other districts are shown in grey for context",
    x = "Epidemiological week",
    y = "Number of cases"
  ) +
  theme_minimal(base_size = 14) +
  theme(panel.grid.minor = element_blank())

28. Principle 4: Label clearly

Good labels reduce mental work.

Use:

  • A title that states the insight.
  • A subtitle that adds context.
  • Axis labels with units.
  • Direct labels where possible.
  • Source notes when using external data.

Avoid vague titles like Results, Graph, or Analysis.

29. Direct labels instead of legends

last_points <- cases |>
  group_by(district) |>
  filter(week == max(week)) |>
  ungroup()

cases |>
  ggplot(aes(x = week, y = cases, color = district)) +
  geom_line(linewidth = 1.2) +
  geom_text(
    data = last_points,
    aes(label = district),
    hjust = -0.1,
    show.legend = FALSE
  ) +
  coord_cartesian(xlim = c(1, 13)) +
  labs(
    title = "Direct labels make the groups easier to follow",
    x = "Epidemiological week",
    y = "Number of cases"
  ) +
  theme_minimal(base_size = 14) +
  theme(
    legend.position = "none",
    panel.grid.minor = element_blank()
  )

30. Storytelling with structure

A good data story often follows:

  1. Setup.
  2. Conflict or change.
  3. Insight.
  4. Implication.
  5. Action.

This structure helps audiences remember the message.

31. Principle 5: Annotate the insight

Annotations help the audience see what you see.

Good annotations:

  • Point to a specific data feature.
  • Explain why it matters.
  • Stay short.
  • Do not cover the data.

32. Annotated plot

cases |>
  mutate(is_focus = district == "North") |>
  ggplot(aes(x = week, y = cases, group = district)) +
  geom_line(aes(color = is_focus), linewidth = 1.3) +
  geom_point(aes(color = is_focus), size = 2) +
  annotate(
    "text",
    x = 7,
    y = 66,
    label = "Peak: 60 cases",
    color = "#3F9C35",
    fontface = "bold"
  ) +
  annotate(
    "segment",
    x = 7,
    xend = 7,
    y = 63,
    yend = 60,
    color = "#3F9C35",
    linewidth = 0.8,
    arrow = arrow(length = unit(0.15, "cm"))
  ) +
  scale_color_manual(
    values = c("TRUE" = "#3F9C35", "FALSE" = "grey75"),
    guide = "none"
  ) +
  labs(
    title = "North district reached its peak in week 7",
    subtitle = "The rapid rise suggests earlier transmission acceleration",
    x = "Epidemiological week",
    y = "Number of cases"
  ) +
  theme_minimal(base_size = 14) +
  theme(panel.grid.minor = element_blank())

34. Principle 6: Show uncertainty honestly

When data are estimates, show uncertainty.

Examples:

  • Confidence intervals.
  • Prediction intervals.
  • Ranges.
  • Error bars.
  • Transparent bands.

A clean uncertainty display builds trust and prevents overclaiming.

35. Practical 2: add uncertainty

weekly_summary <- cases |>
  summarise(
    mean_cases = mean(cases),
    lower = mean(cases) - sd(cases),
    upper = mean(cases) + sd(cases),
    .by = week
  )

weekly_summary |>
  ggplot(aes(x = week, y = mean_cases)) +
  geom_ribbon(
    aes(ymin = lower, ymax = upper),
    fill = "#00507F",
    alpha = 0.18
  ) +
  geom_line(color = "#00507F", linewidth = 1.2) +
  labs(
    title = "Average weekly cases with variation across districts",
    x = "Epidemiological week",
    y = "Mean cases"
  ) +
  theme_minimal(base_size = 14)

36. From chart to story

A chart becomes a story when it has:

  1. A clear audience.
  2. One main message.
  3. Evidence from the data.
  4. A meaningful interpretation.
  5. A possible action or decision.

Story frame:

Among [audience/group], [main finding] during [period], suggesting [interpretation or action].

38. Rapid-fire review

True or false?

  1. More colours always improve a graph.

  4. Annotation can guide audience attention.

39. Key takeaways

  • Start with the message, not the chart.
  • Match the chart type to the question.
  • Use position, length, and order for clear comparison.
  • Remove clutter.
  • Use colour and annotation to guide attention.
  • Show uncertainty when data are estimates.
  • Write titles that explain the insight.
